Overview

Executive Summary

This document proposes a voice-enabled AI assistant that supports infrastructure inspectors during the on-site phase of WSV-regulated waterway inspections. The system bridges the gap between damage suspicions pre-identified in the office and the physical reality found on site — using audio capture, AI extraction, and structured human review to enrich and validate damage records in InfraCloud.

The proposal covers the system's core workflow, functional requirements, AI capabilities, and non-functional constraints. It explicitly surfaces the architectural and UX decisions that remain open and require resolution before implementation begins.

The objective is not to replace damage creation, but to bridge the gap between pre-inspection assumptions and on-site reality through structured, low-friction validation and enrichment.

Section 1

Core Workflow

The inspection lifecycle has two phases. Phase 1 is handled by InfraCloud today and is out of scope. This system targets Phase 2 — the on-site inspection — where inspectors validate, enrich, and create damage records.

Operating Modes

The system must function in two modes, switching automatically based on connectivity. Mode 2 is always available as a fallback and safety net — no AI availability is required for field capture.

Section 3

Functional Requirements

Eight functional requirements define the system's capabilities. Those with unresolved design decisions are marked with an open decision indicator.

FR1
Load inspection context ⚠
Load active inspection and full suspicion list (IDs, locations, boundaries, existing attributes) on session start. Optionally render a lightweight 3D model for spatial context. Display progress: "12 of 45 suspicions validated."
FR2
Reference a suspicion ⚠
Select suspicion → record voice → attach photos → advance. If the lightweight 3D model is available, selection can be spatial (tap on model); otherwise sequential or manual list. Tie-breaking logic is an open decision.
FR3
Create a new damage ⚠
Initiate → voice description → attach photos → set location. Location capture method is an open decision.
FR4
Data storage & sync
Store all audio and photos locally with offline-first architecture. Sync to cloud AI services when connectivity is restored.
FR5
AI extraction pipeline ⚠
Run STT → field extraction → WSV catalog mapping on captured audio. Whether this runs inside InfraCloud or as a separate service is an open architectural decision.
FR6
Delta visualisation
Side-by-side before/after comparison per field, with confidence scores. Low-confidence fields are highlighted. Boundaries and coordinates are never modified.
FR7
Manual validation
Inspector reviews, adjusts incorrect attributes, and confirms. System sends structured update payload to InfraCloud. Never auto-submits.
FR8
Continuous learning
Inspector corrections feed back into the AI pipeline for iterative accuracy improvement. GDPR compliance required before real recordings are used for training.

FR1 in detail — Lightweight 3D model on device

As part of loading the inspection context, the field app may pre-load a lightweight decimated version of the InfraCloud 3D model. This gives the inspector spatial awareness of suspicion locations on device — without a full browser session — and supports two downstream use cases: spatial suspicion selection in FR2, and approximate new damage positioning in FR3.

  • FR2 relevance: suspicion markers overlaid on the simplified mesh allow spatial tap-to-select, reducing reliance on list navigation when multiple suspicions cluster on the same wall section.
  • FR3 relevance: inspector can tap the model to set an approximate location for a newly discovered damage, later refined to a precise coordinate during office review.
Open decision — Feasibility of lightweight 3D rendering on mobile / tablet

A feasibility assessment is required before this can be scoped. Key open questions — all with direct timeline implications:

  • Hardware performance: InfraCloud models are GPU-intensive (sonar point clouds, photogrammetric meshes). An LOD or decimated variant must achieve acceptable frame rates within mobile memory and battery constraints for a full inspection shift.
  • Data volume and offline pre-load: the model must be available on device before going on site. File sizes for harbour assets can be large — partial tile loading or local hotspot streaming must be assessed per asset type.
  • Coordinate alignment: suspicion markers must render correctly against the model using InfraCloud's WGS84 coordinate system. This must be confirmed against InfraCloud's export capabilities.
  • Timeline: 3D rendering on mobile is significant engineering effort. Current position — Iteration 2 delivers list-based selection; lightweight 3D rendering is deferred to Iteration 3 pending feasibility sign-off.

FR2 in detail — Suspicion selection and capture flow

Fig 3. FR2 — Four-step suspicion handling flow. Step 4 is always non-blocking.
Open decision — Suspicion selection method

Three selection methods are under consideration. The primary method and tie-breaking logic must be decided before UX design begins. If the lightweight 3D model (FR1) is available on device, spatial selection becomes a fourth option — see FR1 feasibility assessment.

  • Sequential (primary candidate): system presents next suspicion by proximity. Requires tie-breaking rule for equidistant cases and phase-boundary switching logic.
  • Manual UI selection: inspector picks from a list. Lower friction for adoption, but less efficient at scale.
  • AR-based targeting: inspector points at the physical wall to select a record. Highest usability but highest implementation complexity.

FR3 in detail — New damage location capture

When an inspector discovers a damage with no pre-existing suspicion, a new record must be created on-site. The location capture step has two candidate approaches. If the lightweight 3D model is available on device, a third option — tap-to-position on the model — becomes viable (see FR1).

Open decision — Location capture for new damages
  • Option A — Stationing reference (verbal): inspector narrates position (e.g. "mark 40, halfway to 50"); office reviewer places the point on the 3D viewer before submission. Concern: requires two-step UX if AI service is separate from the 3D viewer.
  • Option B — Placeholder coordinate: AI service creates the record in InfraCloud with a "location pending" flag; engineer sets the precise coordinate in the InfraCloud viewer as a subsequent step. Recommended as the pragmatic MVP path.

Section 4

System Architecture

The system is structured across three independent layers — field capture, AI processing, and office review — with InfraCloud acting as the system of record at both ends. Layers communicate asynchronously so that field work is never blocked by network or AI availability.

Fig 3. Three-layer architecture. Field and AI layers feed the office review workspace; only human-confirmed payloads are written back to InfraCloud.
Offline-first — field app stores all audio and photos locally and syncs when connectivity is restored. No data is lost during outages.
AI is assistive — the workflow never waits for or depends on AI processing. AI accelerates review but is not on the critical path.
Human-in-the-loop — the review workspace requires explicit inspector approval before any data is written to InfraCloud.
Separation of concerns — field app handles capture, AI service handles processing, InfraCloud remains the single system of record.
Section 5

AI Capabilities

The AI pipeline goes beyond transcription. It must classify inspector intent, map natural German speech to structured WSV catalog fields, and detect differences between existing record values and newly captured voice input.

Language scope — Iteration 1: German only. All audio input, transcription, and field extraction targets German-language inspection recordings exclusively. English language support is explicitly out of scope for this iteration and will be addressed in a later phase alongside the Netherlands market expansion. The Whisper STT node is hardcoded to language: de and no language detection or fallback logic is included.

Required intent classification

Intent Description Example utterance (German)
VALIDATE_DAMAGE Inspector confirms the suspicion is a real damage "Schaden bestätigt, Längsriss, Klasse drei."
REJECT_DAMAGE Inspector marks the suspicion as incorrectly detected "Das ist kein Schaden, bitte als fehlerhaft markieren."
UPDATE_FIELD Inspector corrects or overrides an existing attribute "Das ist eigentlich Stahlkorrosion, kein Betonschaden."
ADD_MEASUREMENT Inspector provides a dimension value "Länge dreißig Zentimeter, Breite zwei."
CREATE_NEW_DAMAGE Inspector describes a damage with no prior suspicion "Neuer Schaden, Kantenabbruch, Schadensklasse zwei."

WSV catalog field mapping

The inspector speaks German. Extracted values are mapped to the German WSV catalog strings stored in InfraCloud's backend. The AI must understand that spoken terms like "Längsriss trocken" map to "Risse | Längsriss (trocken)", and that any steel corrosion type maps to "Stahl | Verrostet" regardless of the specific corrosion form described.

Difference detection

When a voice input differs from an existing record value, the system must identify the difference, surface it in the delta view, and propose an update — without auto-applying the change.

Section 6

Open Decisions

The following design decisions must be resolved before implementation begins. Each has a stated impact on architecture, UX, or compliance.

1
Suspicion selection — primary method and tie-breaking

Must choose between sequential (proximity-based), manual UI selection, and AR-based targeting as the primary mode. Sequential mode requires explicit tie-breaking for equidistant suspicions and phase-boundary switching logic.

High · UX
2
New damage location capture method

Option A (stationing reference + 3D viewer placement) or Option B (placeholder coordinate + "location pending" flag). Option B recommended as the MVP path. Decision must align with FR5 architectural choice.

High · UX
3
AI pipeline placement — within InfraCloud or separate service

Determines whether the 3D viewer is available during office review (inline = yes, separate = no). This directly drives the FR3 location UX decision above.

High · Architecture
4
Skip/mandatory decision reconciliation

Decisions are mandatory per suspicion, but UX must remain non-blocking. Proposed rule: "skip" is allowed mid-session; session cannot be marked complete if any suspicions remain in skipped state — inspector is warned at session close.

Medium · UX
5
Lightweight 3D model rendering on mobile — feasibility assessment required

Rendering a simplified InfraCloud 3D model on the field device would enable spatial suspicion selection and on-device damage positioning. Feasibility must be assessed across hardware performance, offline data volume, coordinate system alignment, and timeline impact before this can be scoped. Likely deferred to Iteration 3.

High · Architecture
6
GDPR legal basis for FR8 continuous learning

Voice recordings of named engineers are potentially identifiable personal data. Using them for model training requires explicit consent and a documented legal basis under GDPR Art. 6. Must be resolved before any real recordings are used for training.

High · Compliance
Section 7

Non-Functional Requirements

  • Offline-first field operation

    Full data capture (audio, photos, decisions) without network connectivity. Atomic writes per capture event to support session crash recovery.

  • Optional near-real-time AI assistance

    When connectivity and pipeline performance allow, process audio in near-real time and surface field suggestions on-device during inspection.

  • Deferred AI processing

    If real-time processing is unavailable or incomplete, queue captured data for later processing without blocking field workflow. Both modes converge on the same confirmation flow.

  • Non-blocking UX

    Inspector must always be able to advance to the next suspicion or create a new damage record without waiting for AI processing to complete.

  • Audio quality resilience

    STT pipeline must handle outdoor conditions: wind, waves, boat engine noise. "Low confidence — please re-record" fallback required when transcript confidence falls below threshold.

  • Session crash recovery

    Local store must support atomic writes per capture event. In the event of device failure or app crash, the session must be fully recoverable without data loss.

Section 8

Out of Scope

The following items are explicitly excluded from this proposal. They may be addressed in subsequent iterations.

Underwater inspection workflows — the diver-to-topside relay model requires a fundamentally different interaction pattern and is deferred to a later iteration.
English language support — all audio capture, transcription, and AI extraction in this iteration targets German-language inspection recordings only. English and Dutch language support will be introduced in a later iteration alongside international market expansion.
Authentication and authorisation — Keycloak integration, the InfraCloud user model, and role-based access rights are out of scope. The AI system inherits the existing InfraCloud session and permission model without modification.
Pre-inspection Phase 1 — the 3D model review, suspicion creation, coordinate setting, and boundary drawing remain unchanged and out of scope.
Real-time video analysis or AR damage overlay — AR-based suspicion selection is under consideration (Open Decision 1) but AR spatial overlay of damage boundaries is out of scope for this proposal.
