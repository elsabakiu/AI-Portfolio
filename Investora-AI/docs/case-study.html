Case Study · AI Product Engineering

Investora AI

A personalized investment-signal engine that turns market, fundamentals, and news data into a small, ranked list of weekly priorities for retail investors.

The core idea is unglamorous: most of the work is deterministic Python. The LLM is one step in a 17-node graph, called only when something has to be framed in natural language. Everything else — scoring, ranking, anomaly detection, personalization, mismatch detection — happens in code that can be tested with a numeric diff.

What it does
Runs a 17-stage analysis pipeline over a configurable ticker universe, scores every name on quality and momentum, detects volume spikes and score deltas, retrieves prior-week evidence from a vector store, asks the LLM for a structured framing, ranks the resulting signals against each user's profile, and delivers the result as a personalized dashboard, an email digest, and Telegram alerts.
How it does it
FastAPI exposes the orchestration. LangGraph runs the pipeline. SQLite holds users, profiles, watchlists, alerts, runs, signals, bundles, and an API-call budget. n8n schedules the runs and fans out the deliveries. The frontend is React + TypeScript, fed entirely by typed JSON contracts.
Positioning — Not Bloomberg (too much). Not Robinhood-style cards (too generic). Not Seeking Alpha (too retail-analyst). The unoccupied space is profile-aware, signal-level, weekly-cadence — and that's the slot Investora is built for.

01 · Problem framing

The problem Investora is built around

A retail investor with a watchlist, a job, and an opinion does not have an information problem. They have a prioritization problem. The signals exist; they are just buried under a feed designed to maximize attention. Twitter shouts. Bloomberg drowns. Robinhood flattens everything to one-line momentum cards. Seeking Alpha gives long-form theses with no connection to what you actually hold.

The user need that fell out of that observation, in plain language: "On a Sunday evening, give me three things from my watchlist worth thinking about this week, plus one or two opportunities I would not have found on my own — and tell me when something I already hold has stopped fitting the profile I gave you."

That sentence is the spec. Every architectural decision below either serves it or gets cut.

Founding constraint
Calibrated relevance, not accuracy. Friction over flattery.
A recommender that only confirms what the user already believes is a confirmation-bias engine. Investora's most distinctive feature — mismatch alerts — exists to tell users when their holdings have drifted away from their stated profile. No user asks for this. It is the feature the product is most opinionated about.
02 · Capabilities

What Investora does (the user-visible surface)

What the user actually sees, in the order they encounter it:

Capability · 01
Onboarding that captures the constraints, not just the preferences.
A new user registers, then walks through risk tolerance (low / medium / high plus a 0-100 slider), interests (tech / crypto / energy / forex / commodities), time horizon (short / medium / long), constraints (no_crypto, ESG, max_20pct), preferred asset classes, base currency, and an optional Telegram chat ID. None of these are decorative — every field changes how downstream signals are scored and filtered.
Capability · 02
A personalized weekly bundle, split into three buckets.
After each run, the user's dashboard shows a UserReportBundle: portfolio pulse (1-day and 1-week watchlist performance, market regime, risk-alignment verdict), top conviction (the three highest-fit signals across every bucket), watchlist signals (sorted by urgency then fit), discovery signals (opportunities outside the watchlist, gated on confidence + profile fit), and mismatch alerts (high-severity signals that clash with the user's risk tolerance). Each signal carries a fit score, an action frame (Review, Monitor, Trim Risk), an urgency level, a narrative, and a list of risk flags.
Capability · 03
Live market data on demand — no run required.
Independent of the analysis pipeline, the app exposes always-on market endpoints: NYSE open/closed status, real-time quotes for arbitrary tickers, OHLCV charts (1D / 5D / 1M / 3M / 6M / 1Y), a merged intraday + fundamentals snapshot, recent news headlines, top gainers/losers from a ticker set, ticker autocomplete search, and a 2–3 sentence cached AI view per ticker. These are what the dashboard, the Watchlist page, and the per-ticker Stock Analysis page consume.
Capability · 04
User-defined price alerts that fire independent of the agent loop.
Users create alerts of three types: price_above, price_below, daily_move. They are checked inside the LangGraph pipeline (not as a separate cron) so they share the same data fetch as the analysis run. When triggered, they post to an n8n webhook for delivery. Alert state (active / triggered / disabled) is persisted in SQLite and editable from the Alerts page.
Capability · 05
A live progress stream — no black-box runs.
When the user clicks Run Analysis, the request hits /run-analysis-stream and the graph starts in a background thread. Each completed node is pushed to the browser as a Server-Sent Event. The dashboard renders a progress bar and the current stage name (Compute Scores, Synthesize Evidence, etc.). When the pipeline finishes, the React Query cache for the dashboard, run history, and personalized signals is invalidated automatically. The user sees the new bundle without a refresh.
Capability · 06
Push delivery — Telegram for urgency, email for the weekly digest.
Two delivery channels, deliberately different. Telegram is reserved for high-urgency signals from the personalized bundle, formatted as Markdown with a colored direction emoji and an action frame. The weekly email digest is HTML, fired only on the Monday-morning full run, gated per-user on a weekly_email_digest profile flag. Both go through n8n webhooks; both fail soft.
03 · System architecture

How the pieces fit together

The system has four planes and one orchestration boundary. The reasoning plane runs the analysis. The data plane holds users, profiles, runs, signals, and bundles. The serving plane is the FastAPI app that exposes both the agent endpoints and the always-on market endpoints. The delivery plane is n8n. The orchestration boundary is the typed JSON contract between Python (Pydantic) and TypeScript (Zod) — the same shape on both sides, validated on both sides.

The reasoning pipeline writes to SQLite. Delivery reads from SQLite via n8n webhooks. Delivery failures never propagate back upstream — they are caught and logged at the graph node level.

Frontend

Single-page React app built with Vite. Routing is split between public pages (Landing, Login, Register, Terms, Privacy, Disclaimer) and a protected shell wrapping Dashboard, Watchlist, Stock Analysis, Alerts, Strategy, and Profile. Code-splitting is used for every protected route. State is held in three places: a UserContext for session and watchlist, React Query for everything fetched, and localStorage for the cached session payload (rehydrated on mount, validated against /auth/me when a token is present). Visual identity is dark navy with electric cyan and neon green accents — a fintech-terminal aesthetic, not a Robinhood pastel one.

Backend

FastAPI with three middleware layers: a CORS layer that allows configured origins plus any localhost port for dev, a correlation-ID middleware that assigns a UUID to every request and echoes it back in X-Correlation-ID, and a redaction filter on logging so secrets never appear in stdout. The app boots a small set of long-lived services on import — AnalysisService, PersonalizationService, NotificationService — backed by repository classes that mediate every SQLite read and write.

04 · The reasoning pipeline

The 17-node LangGraph pipeline, walked end-to-end

The graph is compiled once per process, cached as a singleton, and executed per run with a recursion limit. State is a TypedDict threaded through every node. Every node is wrapped in a timer that records its duration both into the run state and into a global metrics snapshot, so node-level latency is observable from /debug/metrics without any external tracer.

The first three nodes form a classic ReAct loop — a planner picks the next data action, an executor runs it, the planner replans — until every (ticker, action) pair has been satisfied. The planner is an LLM call when running live and a deterministic first-pending picker when running against mocks. Nodes 4 onward are linear.

01
init_state
Generate run ID and run date. Resolve the ticker universe (request override → mock universe file → STOCK_UNIVERSE env → 10-ticker default). Load every user profile from SQLite, collect their watchlists, and ask the budget manager to prioritize the universe so user-watchlist tickers are processed first when the daily API budget is tight. Initialize the empty buckets in graph state.
02
plan_next_action
Compute every (ticker, action) pair still missing. In mock mode, pick the first one. In live mode, send the pending list to the OpenAI Responses API with a strict instruction to return JSON of the form {ticker, action, reason}. Validate the choice against the pending set. If the planner errors, fall back to the first pending pair. When nothing is pending, return action=compute and exit the loop.
03
execute_tool_action
Run one MCP tool — market, fundamentals, or news — for the selected ticker. The market tool batches when possible; fundamentals and news fan out under a thread pool with a configurable parallelism. Every call is gated by the budget manager (per-provider, per-day cap) and writes back into per_ticker_data. Failures land in errors and the ticker into failed_tickers. Loop returns to plan_next_action.
04
compute_scores
For every eligible ticker (has both market and news data), compute a quality score from fundamentals (profitability, balance-sheet health, growth) and a momentum score from the 5-trading-day price return. Normalize both to a 0–10 range across the universe. Combine them into an overall score using a configurable quality/momentum weighting (defaults to roughly 55/45). Persist scores into graph state.
05
detect_anomalies
Compare the current run against the previous run loaded from the event store. Flag two anomaly classes: volume spikes (current volume materially above the rolling average) and score deltas (a ticker whose overall score moved meaningfully versus the prior run). Anomalies become first-class signal events with their own severity.
06
retrieve_rag_context
For each ticker in the evidence target set (top-N by score, plus any user-watchlisted ticker that fell outside top-N), embed a query like "{TICKER} catalysts, trend confirmation, risk factors" with OpenAI embeddings, then query Pinecone with a ticker filter and a recency cutoff. Results land in per_ticker_rag_context. Skipped entirely on fast-mode runs.
07
synthesize_evidence
For each evidence-target ticker, build a structured bundle (price summary, fundamentals percentages, last 5 headlines, RAG matches, scores) and ask the synthesis model for a JSON response with quality_narrative, momentum_narrative, news_catalyst, and risk_factors. The system prompt bans buy/sell language. Calls fan out under a thread pool. Skipped on fast-mode runs.
08
emit_signals
Build the canonical SignalEvent list from scores plus per-ticker synthesis, then merge in the anomaly events from node 05 (deduping by ID). Each event carries ticker, signal_type, direction, severity, confidence, score, narrative, route, and run linkage.
09
personalize_signals
For every loaded user profile, build a UserReportBundle. Each signal is rated against the user along three axes (watchlist relevance, profile fit, risk mismatch) and combined into a fit score. Signals are routed to the watchlist or discovery bucket; discovery is gated on minimum confidence + profile fit + fit score thresholds. Mismatch alerts are emitted when a high-severity signal fires for a low-risk user. Bundles are persisted immediately to user_report_bundles.
10
check_user_alerts
Walk every active user-defined alert in the user_alerts table, compare the threshold against the freshly fetched market data for that ticker, and collect the triggers. Marks fired alerts in SQLite and accumulates a triggered_user_alerts list for delivery.
11
post_alerts
For each user with a Telegram chat ID and notifications enabled, dedupe and forward the high-urgency signals from their bundle to the ALERT_WEBHOOK_URL n8n endpoint. Triggered user-defined alerts go to a separate webhook. Both fail soft — exceptions are caught, recorded into the run's errors, and the pipeline continues.
12
post_candidates
Only fires on weekly full runs (trigger_weekly_digest=true, scope=full). For each user with the weekly-digest flag set and a non-empty email, build a weekly digest payload from their bundle and forward it to the CANDIDATE_WEBHOOK_URL.
13
assemble_report_json
Build the typed ReportPayload — top opportunities, strategy breakdown, insights, data sources, system metadata, anomaly signals, tool errors. This is the legacy report shape served by /latest-report and consumed by the Strategy page and the email digest pipeline.
14
assemble_markdown
Render the JSON report as Markdown for the email digest path. Pure formatting — no recomputation.
15
post_to_n8n
POST the full JSON report to N8N_WEBHOOK_URL for archival and downstream automation. Skipped when the run was started with no_post=true or when SKIP_N8N_POST is set.
16
persist_report
Write the JSON report to data/reports/{run_date}.json. The file-based history is what /latest-report reads.
17
persist_snapshot
Insert one row into analysis_runs and one row per signal into signal_events. Then post a small monitor event (run status, error count, signal count) to the n8n run-monitor webhook so a failed weekly run can page somewhere.
Design note
One LLM call line, two LLM call sites.
There are exactly two places the pipeline reaches for an LLM: the planner (one call per loop iteration to pick the next data action) and the synthesizer (one call per evidence-target ticker to produce the structured narrative). Everything else — scoring, anomaly detection, fit calculation, ranking, mismatch detection, bundle assembly — is deterministic Python that can be regression-tested with a numeric diff. Cost and latency are predictable; the LLM's job is framing, not arithmetic.
05 · Scoring, ranking, personalization

How a raw signal becomes a personalized recommendation

Quality score

A composite of three fundamentals dimensions: profitability (return on equity, operating margin), balance-sheet health (debt-to-equity, lower is better), and growth (revenue growth and EPS growth). Each component is bounded into a 0–10 range, then blended with fixed weights — profitability is given the most weight, balance-sheet and growth roughly equal. Missing inputs default to a neutral 5.0, so a ticker is never penalized for absent fundamentals data; it just doesn't get a high score.

Momentum score

The 5-trading-day return of the closing price. Strictly price-based — no implied-volatility, no relative-strength, nothing fancy. The intent is for the score to be cheap to compute, easy to explain, and resistant to provider drift. After the raw return is computed, scores are min-max normalized across the universe so the ranking is always interpretable as "best of this week's run."

Overall score

A weighted blend of quality and momentum. The default weighting biases slightly toward quality, which expresses the product's intentional preference for medium-horizon over swing-trade signals. The weights are configurable from the environment so they can be tuned without a code change.

Personalization fit

Each emitted signal is rescored per-user along three axes:

  • Watchlist relevance — binary. Either the ticker is on the user's watchlist or it isn't. Watchlist signals get the highest priority, always.
  • Profile fit — qualitative match between the signal's metadata (asset class, sector, themes) and the user's stated interests, preferred asset classes, and time horizon. A signal in a sector the user marked as an interest scores higher; a signal whose asset class is on the user's preferred list scores higher; a momentum signal for a user with a short horizon scores higher and a momentum signal for a user with a long horizon scores lower. Hard constraints (e.g. no_crypto) zero the fit out completely — these are enforced in code, not in the prompt.
  • Risk mismatch — a penalty applied when the signal severity is high but the user's stated risk tolerance is low. This is what feeds the mismatch-alerts surface.

The three axes combine into a single fit score in roughly the 0–1 range — watchlist relevance and profile fit pull it up, risk mismatch pulls it down. Watchlist signals are sorted by urgency then fit; discovery signals are sorted by fit alone, and gated on minimum confidence + profile fit + fit score thresholds before they are shown to the user. The thresholds live in environment variables so they can be tuned without a deploy.

Action frame

Every signal carries one of three labels: Trim Risk (held in watchlist, severe, direction down), Review (held in watchlist, severe), or Monitor (everything else). The frame is what the dashboard surfaces — not "buy" or "sell." The product's stance is that users are responsible for the trade; Investora is responsible for the prioritization.

Market regime & risk alignment

Two derived verdicts that sit at the top of the dashboard. Market regime is computed from the average momentum across the universe — Risk-On, Neutral, or Risk-Off. Risk alignment is computed from the user's watchlist — when a low-risk user holds a watchlist whose average overall score is unusually high, the dashboard surfaces an "Off-Profile" or "Caution" verdict. Both are deterministic.

06 · Data layer

What gets stored, and where

State lives in a single SQLite file at langgraph/data/investora.db, opened in WAL mode with foreign keys on. Eight tables, all created idempotently from one DDL block on first connect. SQLite was a deliberate MVP-stage choice — it's portable, fast, and the schema is intentionally simple enough to migrate to Postgres without a rewrite when the user count justifies it.

users
Account credentials. PBKDF2-SHA256 password hashes with a per-user salt; usernames are normalized lowercase and uniquely indexed.
PK user_id · UNIQUE username
user_profiles
A single JSON blob per user, holding risk tolerance, interests, horizon, constraints, preferred asset classes, base currency, digest preferences, Telegram chat ID, and any manually entered positions.
PK user_id · profile_json
user_watchlists
One row per user; the tickers are stored as a JSON array of uppercase strings. Read by every analysis run during init_state and joined into profile loads via a single LEFT JOIN to avoid an N+1 read.
PK user_id · tickers_json
user_alerts
User-defined price alerts. Each row carries a ticker, a condition (price_above / price_below / daily_move), a numeric threshold, a status, and timestamps.
PK id · idx user_id · idx status
analysis_runs
One row per pipeline execution: scope (full / fast), tickers run, score map, failed tickers, error count. The historical record that the dashboard, the strategy page, and the run-history dropdown all read from.
PK run_id · run_date
signal_events
The atomic output unit of a run. One row per signal with ticker, type (quality / momentum / news_catalyst / risk_flag / volume_spike / score_delta), direction, severity, confidence, score, narrative, and route. Indexed for ticker-history lookups by ticker + date.
PK id · FK run_id · idx ticker+date
user_report_bundles
Per-user, per-run snapshots — the JSON blob the dashboard renders. Stored as the bundle is built (in node 09) so a partial pipeline failure later in the run still leaves the user's dashboard correct.
PK (user_id, run_id) · idx user_id+date
api_budget_log
Per-day, per-provider call counter. The budget manager increments this on every external call; when a per-day cap is hit, subsequent calls are skipped and the affected tickers are marked failed for the run. Prevents a runaway loop from blowing through a free-tier quota.
PK (date, provider)
07 · Integrations & delivery

External systems and the n8n boundary

Data providers

Four live providers, each with a deterministic mock counterpart toggled by USE_MOCK_DATA. Marketstack and yfinance back market data; yfinance covers the always-on dashboard endpoints (quotes, charts, snapshots, search, top-movers) and Marketstack covers the pipeline's batch market fetch. Financial Modeling Prep backs fundamentals. Finnhub backs news. Pinecone backs RAG retrieval; the embeddings come from OpenAI's text-embedding-3-small. OpenAI drives the planner, the synthesizer, the embeddings, and the on-demand AI-view summaries on the Stock Analysis page.

n8n workflows

Six workflows, each a small JSON export under n8n/:

cron-weekly-full.json
Monday 06:00 UTC. POSTs /run-weekly with a Bearer secret. Triggers the full digest path.
cron-daily-fast.json
Daily fast run. Skips synthesis; refreshes scores, anomalies, and personalized bundles.
telegram-alerts.json
Receives the alert webhook payload, splits it per-user, formats a Markdown message, sends to Telegram.
weekly-candidates-email.json
Receives the weekly digest payload and renders the HTML email per user.
run-monitor.json
Receives the post-run monitor event. Pages on errors and high failure counts.
auth.json
Legacy auth flow workflow, kept for reference.

The pattern is consistent: the Python pipeline never talks directly to Telegram or to an SMTP server. It talks to a webhook. n8n owns the credential, the formatting, and the retry. If a webhook URL is unset, the corresponding delivery silently no-ops — the analysis still completes. This is the single most important reliability decision in the system.

Deployment

Backend on Render (Python service, autoscaling off, env vars injected from the dashboard). Frontend on Vercel (Vite static build, SPA rewrite). The full deployment config is checked in under deployment/render.yaml and deployment/vercel.json. Sentry is wired on both ends, opt-in via DSN — initialization failures are swallowed so a broken DSN never crashes the server.

08 · Reliability and observability

What protects the pipeline from itself

A weekly cron job that touches four external providers, an LLM, a vector store, and three webhooks has a lot of failure modes. The reliability stack is small but specific:

Reliability · 01
Deterministic mock mode for every provider.
Every MCP tool has a mock counterpart. Setting USE_MOCK_DATA=true flips the entire pipeline to deterministic fixtures, including a 100-ticker S&P-100 mock universe with sector metadata. Tests, local dev, and CI all run end-to-end without touching a single paid API.
Reliability · 02
Per-provider, per-day API budget.
A budget manager intercepts every provider call, increments a counter in api_budget_log, and refuses calls that would breach the daily cap. Watchlist tickers are prioritized in the universe order so a budget-starved run still produces a useful dashboard for paying users. When the budget runs out mid-run, affected tickers are added to failed_tickers rather than crashing the run.
Reliability · 03
Run limiter to bound concurrency.
A single in-process semaphore caps concurrent runs at a configurable limit, with a bounded queue and a wait timeout. New requests get a 429 with a structured message when the queue is full. Prevents a click-mash on the dashboard's Run Analysis button from spawning a thundering herd of OpenAI calls.
Reliability · 04
Delivery isolation — every outbound call fails soft.
Every post_* node wraps its HTTP call in a try/except, catches every exception, appends to the run's errors, and continues. A failing Telegram webhook does not break the email digest; a failing email digest does not break the persisted dashboard. The reasoning layer never knows whether delivery succeeded.
Observability · 01
Per-node timings and provider call counts, exposed.
Every node is wrapped in a timer that records duration both into the run state and into a global metrics snapshot. /debug/metrics returns the full snapshot — node timings, provider call counts, recent run summaries — without an external tracer. Cheap, in-process, and good enough for the current scale.
Observability · 02
Correlation IDs and a redaction filter on logs.
Every request gets a UUID echoed back in X-Correlation-ID. The redaction filter is installed on the root logger before any provider call, so API keys and user passwords never appear in a structured log line — even when an exception trace would otherwise leak them.
Observability · 03
Contract tests on the user-facing endpoints.
The test suite under langgraph/tests/ covers the mock fast-path graph integration plus three contract tests: /run-analysis-stream, /user/{id}/dashboard, and /user/{id}/personalized-signals. The same shape is validated again on the frontend with Zod, so a Python-side schema change can't ship silently.
09 · Where the seams are

Known limitations — what would break first at 10× scale

What works today

  • End-to-end weekly + daily cron pipeline running against live providers.
  • Personalized dashboards, watchlist signals, discovery signals, mismatch alerts.
  • Streaming run progress to the UI via Server-Sent Events.
  • User-defined price alerts checked inside the same pipeline as analysis.
  • Telegram + email delivery via n8n webhooks, with soft-fail isolation.
  • Mock-mode parity — every provider has a deterministic stand-in.
  • Typed report contract — Pydantic on the server, Zod on the client, both validated.

Known gaps

  • SQLite single-writer ceiling. WAL helps, but the database is one file. Above a few concurrent writers, queue contention will start to show. The schema is intentionally simple enough to migrate to Postgres without a rewrite.
  • Synthesis quality is provider-bound. When OpenAI is slow or rate-limited, full runs degrade gracefully (skip_synthesis fallback) but the user-visible quality drops. There is no second-source LLM.
  • No formal eval harness. No held-out user-labelled "was this signal useful?" dataset. There is no measurement of whether mismatch alerts actually improve user outcomes — that test is designed and not yet run.
  • Frontend bundle size. The initial JS payload is large; per-route code-splitting is in place but further chunking would help time-to-interactive.
  • Compliance hardening. The advisory disclaimer banner is shown on every page, but the product is not yet hardened to the standard a regulated advisory would meet.
  • No real-time path. Not a limitation — a deliberate refusal. Real-time pushes retail investors toward overtrading. Investora's cadence is daily and weekly, by design.
10 · Where this goes

Roadmap — 30 / 60 / 90 days

Next 30 days · adoption + decision quality
Tighten the first-week activation loop. The current onboarding asks for risk profile, interests, and a Telegram chat ID before the first run; the time to first useful insight is what the next iteration will optimize against. Define and track product KPIs for decision quality — relevance, confidence, follow-through — and tighten signal framing so users can compare opportunities with less cognitive effort.
Next 60 days · product-market fit signals
Strengthen personalization depth using behavior feedback loops — what users keep, dismiss, or act on. Improve alert usefulness with better prioritization and cadence controls. Validate retention hypotheses by cohort: new users vs. repeat weekly users.
Next 90 days · scale + commercial readiness
Expand from single-user workflows toward advisor / team use cases. Introduce portfolio-level narratives and scenario views for higher-value decision contexts. Prepare a production-readiness package focused on reliability, governance, and stakeholder reporting — including the move to Postgres and the first iteration of a formal eval harness.
11 · Transferable lessons

What I carry forward to the next AI product

Schema discipline beats prompt discipline
Typed contracts (Pydantic + Zod, the same shape on both sides) did more for quality than any prompt improvement. Once the output shape was guaranteed, the UI became deterministic, the evals became diffable, and the prompts became swappable. This is the rule I will not relitigate.
Use the LLM for the thing only an LLM can do
Investora has 17 nodes and 2 LLM call sites. Scoring, ranking, anomaly detection, fit calculation, mismatch detection — all deterministic Python. The LLM frames; it does not arithmetic. Cost and latency become predictable; regression testing becomes a numeric diff, not an LLM-as-judge loop.
Constraints live at the product layer, not the prompt
"No crypto," "ESG only," "max 20% position" are enforced in deterministic Python before any LLM call. Prompt-layer constraints are soft suggestions; product-layer constraints are guarantees — which is the thing the user signed up for.
The reasoning layer must survive a broken delivery layer
Every outbound call in Investora is wrapped, caught, and logged — never raised. If Telegram is down, the report is still written and visible in the dashboard. Graceful degradation was wired in before any outage, not after. Probabilistic pipelines have more failure points than deterministic ones, and each has to be decided on, not discovered.
Friction over flattery is a product opinion, not a UX one
Mismatch alerts are the explicit wager. A confirmation-bias system would be more engaging in the short run. Investora is betting against engagement because engagement is the wrong metric for a financial product — people who feel validated over-trade, and over-trading destroys retail returns.
